# 2026-04-24 消息渲染协议设计文档

## 1. 背景

当前 Flutter 前端已经完成基础聊天、流式输出、Markdown 初步渲染，但在真实使用中暴露出一个核心问题：

前端无法稳定判断“模型返回的内容到底应该怎么渲染”。

典型场景：

1. 用户要求“直接返回 Markdown 文本用于测试渲染”。
2. 用户要求“把刚才生成到文件里的 Markdown 内容显示出来”。
3. 模型返回的是可渲染 Markdown 文档。
4. 模型返回的是 Markdown 源码展示。
5. 模型返回的是混合说明文字 + Markdown 文档 + 代码块 + LaTeX。

这些内容在字符串层面相似，但渲染方式完全不同。

如果继续完全依赖前端猜测，会持续出现以下问题：

1. 流式阶段与完成阶段渲染模式不一致。
2. 前端打开历史会话时，需要重新做高成本内容分析。
3. Markdown 文档与 Markdown 源码混淆，导致渲染错误。
4. 文件型产物与直接回答混淆，导致“只写文件不展示正文”。
5. 前端渲染复杂度不断膨胀，后续接入多 Agent、多模态后会更难维护。

因此，这个问题不能继续只靠前端修补，必须引入“后端输出协议 + 前端按协议渲染”的机制。

---

## 2. 今日问题记录

### 问题 1：前端无法知道回答的真实语义

**现象**

当前 `ChatResponse` 只有：

1. `answer`
2. `tool_calls`
3. `memory_hits`

前端拿到的只是一个字符串，不知道它究竟是：

1. 纯文本回答
2. 可直接渲染的 Markdown 文档
3. Markdown 源码
4. 代码回答
5. 大文档预览
6. 某个文件内容的摘录

**结果**

前端只能靠：

1. 字符串长度
2. 行数
3. 代码围栏数量
4. 正则特征

去猜测渲染方式。

这种方式不稳定，也不适合生产级系统。

---

### 问题 2：模型输出格式不稳定

**现象**

模型虽然可以被提示“输出 Markdown”，但仍可能出现：

1. 把整份 Markdown 文档包在外层 ````markdown fenced block` 中。
2. 在正文前后插入说明文字。
3. 本轮生成了文件，却只回复“已经写入文件”，不直接展示正文。
4. 公式不统一，有的用 `$...$`，有的用普通文本描述。
5. 代码块语言标记不统一，甚至缺失。

**结果**

即使前端渲染逻辑完全正确，也会因为模型输出不规范而出现展示异常。

---

### 问题 3：前端历史会话打开成本高

**现象**

当重新进入含复杂 Markdown 的历史会话时，前端需要重新分析每条消息。

**结果**

1. 打开会话可能卡顿。
2. 复杂消息可能在历史态和流式态出现不同的渲染结果。
3. 前端需要保留越来越多的推断逻辑。

---

## 3. 设计目标

本次协议设计目标如下：

1. 让后端明确告诉前端“这条消息应该怎么渲染”。
2. 让模型输出的内容在进入前端前先经过统一归一化。
3. 前端尽量不再猜测回答语义，只负责执行渲染协议。
4. 保持流式链路和历史回放链路使用同一套渲染元数据。
5. 方案必须兼容未来多 Agent、文件型产物、多模态扩展。

---

## 4. 边界划分

### 4.1 后端负责

后端负责“内容治理”和“协议输出”：

1. 识别回答类型。
2. 归一化回答内容。
3. 补充渲染元数据。
4. 告诉前端本条消息是什么，不是什么。

### 4.2 前端负责

前端负责“按协议渲染”：

1. 根据协议选择渲染器。
2. 在必要时做轻量兜底。
3. 不再承担主要语义判断。

### 4.3 模型负责

模型负责“尽量按规范输出”，但不作为唯一可信源。

也就是说：

1. Prompt/skill 约束模型尽量输出标准格式。
2. 后端 normalizer 负责把不稳定结果收敛成稳定协议。

---

## 5. 协议设计建议

建议为每条 assistant 消息增加以下字段。

### 5.1 `answer_format`

用于描述回答的主格式。

建议值：

1. `plain_text`
2. `markdown`
3. `code`
4. `markdown_source`

### 5.2 `render_hint`

用于指导前端具体渲染策略。

建议值：

1. `plain`
2. `markdown_document`
3. `markdown_source`
4. `code_block`
5. `large_document`

说明：

`answer_format` 偏格式语义，`render_hint` 偏前端渲染指令。

### 5.3 `source_kind`

用于说明本条消息内容来源。

建议值：

1. `direct_answer`
2. `generated_document`
3. `file_content`
4. `summary`

### 5.4 `artifacts`

用于说明本轮是否关联文件产物。

建议结构：

```json
[
  {
    "type": "file",
    "path": "test.md",
    "role": "generated"
  }
]
```

`role` 建议值：

1. `generated`
2. `source`
3. `reference`

## 6. 协议示例

### 6.1 直接渲染 Markdown 文档

```json
{
  "answer": "# 标题\n\n正文内容",
  "answer_format": "markdown",
  "render_hint": "markdown_document",
  "source_kind": "direct_answer",
  "artifacts": []
}
```

### 6.2 展示 Markdown 源码

```json
{
  "answer": "```markdown\n# 标题\n\n正文内容\n```",
  "answer_format": "markdown_source",
  "render_hint": "markdown_source",
  "source_kind": "file_content",
  "artifacts": [
    {
      "type": "file",
      "path": "test.md",
      "role": "source"
    }
  ]
}
```

### 6.3 大文档预览

```json
{
  "answer": "# 超长文档\n\n......",
  "answer_format": "markdown",
  "render_hint": "large_document",
  "source_kind": "generated_document",
  "artifacts": [
    {
      "type": "file",
      "path": "report.md",
      "role": "generated"
    }
  ]
}
```

### 8.4 本阶段收益

1. 新消息与历史消息现在都可以共享同一套渲染协议

## 9. 第二阶段已落地内容（2026-04-24）

### 9.1 流式协议补充

新增 SSE 事件：

1. `answer_meta`
2. `answer_meta_reset`

当前行为：

1. 流式正文输出前，后端会基于当前累计内容做一次轻量归一化。
2. 当 `answer_format / render_hint / source_kind` 发生变化时，推送新的 `answer_meta`。
3. 如果本轮先吐出正文、后面又转成 `tool_call`，会先发：
   1. `answer_meta_reset`
   2. `answer_reset`

这样前端流式阶段也可以跟最终态保持同一条渲染决策链。

### 9.2 输出规范已收进后端系统提示

本次没有去散改多个 skill 文件，而是先把输出规范收敛进两处：

1. `ContextAssembler` 的系统提示
2. `AgentRuntime` 的最终补答提示

当前新增的硬约束包括：

1. 渲染型 Markdown 文档不要再包外层 ````markdown``
2. 只有用户明确要求查看 Markdown 源码时才允许使用 ````markdown``
3. 代码回答要使用标准 fenced code block
4. 公式统一用 LaTeX
5. 如果用户要看文件内容，最终答复里要把正文直接回出来，而不是只说“已保存”

这部分不是协议本身，而是模型的输出约束。

## 11. 第四阶段已落地内容（2026-04-24）

### 11.1 文件产物卡片交互

在不继续扩协议的前提下，先补了最小可用交互：

1. 如果 `artifact.path` 是 `file_id:...`

   2. 若命中，会显示 `文件名 (file_id)`
   3. 支持“激活文件”，直接把它加入当前会话上下文
2. 如果 `artifact.path` 是普通路径
   1. 前端显示原始路径
   2. 支持“复制路径”

### 11.2 当前边界

1. 这一步只做“激活上传文件”与“复制路径”
2. 还没有做：
   1. 点击后自动滚动到输入框文件面板
   2. workspace 文件的直接打开/预览
   3. 通过 artifact 反向定位到具体工具调用结果

### 7.1 Markdown 文档输出规范

当目标是“渲染 Markdown 文档”时：

1. 返回可直接渲染的 Markdown 正文。
2. 不要再包外层 ````markdown fenced block`。
3. 标题、列表、代码块、表格按标准 Markdown 语法输出。

### 7.2 Markdown 源码展示规范

当目标是“展示源码”时：

1. 使用外层 ````markdown fenced block` 包裹。
2. 明确这是源码而不是渲染后的文档。

### 7.3 代码输出规范

1. 一律使用 fenced code block。
2. 尽量带语言标记，例如 `python`、`javascript`、`sql`。

### 7.4 公式输出规范

1. 行内公式统一使用 `$...$`
2. 块级公式统一使用 `$$...$$`

### 7.5 文件型结果规范

如果本轮生成了文件，但用户意图是“直接查看内容”，则：

1. 最终答复中仍要包含可展示正文；
2. 不能只回复“文件已写入，请去看文件”。

## 8. 后端改造方案

### 8.1 扩展响应结构

需要扩展：

1. `/api/chat`
2. `/api/chat/stream` 的 `done` 事件
3. 会话历史消息接口（后续建议也补充渲染元数据）

### 8.2 引入回答归一化器

建议在 runtime 最终收口之后、返回 API 之前，引入一个 `AnswerNormalizer`。

职责：

1. 识别回答格式。
2. 解包不合理的外层 ````markdown` 包装。
3. 区分 `markdown_document` 与 `markdown_source`。
4. 给大内容打上 `large_document` 标记。
5. 如果本轮文件是主产物，补充 `artifacts`。

### 8.3 文件相关回答策略

对于涉及文件写入/读取的回答，后端可结合：

1. 本轮工具调用记录
2. 目标文件路径
3. 用户原始问题意图

来辅助判定：

1. 这是文件摘要
2. 还是文件正文展示
3. 还是源码展示

### 8.4 SSE 流式增强

建议后续增加一个早期元数据事件：

1. `answer_meta`

示例：

```json
{
  "answer_format": "markdown",
  "render_hint": "markdown_document",
  "source_kind": "direct_answer"
}
```

这样前端在流式阶段也能提前选 renderer。

第一版如果不想一次改太多，也可以：

1. 先只在 `done` 里补协议字段；
2. 流式阶段继续用轻量 renderer；
3. 完成后按后端元数据切换到最终 renderer。

## 9. 前端改造方案

### 9.1 前端渲染原则

前端收到后端协议后，按如下规则执行：

1. `render_hint=plain` -> 纯文本
2. `render_hint=markdown_document` -> Markdown renderer
3. `render_hint=markdown_source` -> 源码/代码视图
4. `render_hint=code_block` -> 代码块视图
5. `render_hint=large_document` -> 大文档轻量模式

### 9.2 前端兜底策略

若后端未返回协议字段，则前端才使用本地轻量推断逻辑。

这意味着：

1. 推断逻辑从“主路径”降级为“兜底路径”。
2. 前端复杂度可以显著下降。

### 9.3 历史会话一致性

后续历史消息接口若也返回渲染元数据，则：

1. 打开历史会话时不需要重新猜测内容语义。
2. 流式结束态与历史回放态渲染结果可以保持一致。

## 11. 验收标准

完成后至少满足：

1. “直接返回 Markdown 文档”能稳定按文档渲染。
2. “显示 Markdown 源码”能稳定按源码展示，不误渲染。
3. “读取文件后展示内容”不会再只显示文件摘要。
4. 历史会话打开时，不再因为前端重新猜测而卡顿明显。
5. 流式结束态与重新进入会话后的历史态，渲染结果一致。

## 12. 风险与注意事项

### 风险 1：只依赖 Prompt 规范不够

模型不是确定性编译器，不能只靠提示词保证输出稳定。

### 风险 2：协议字段必须全链路贯通

如果只改 `/api/chat`，不改 SSE `done` 和历史消息接口，前端仍会分裂出多套逻辑。

### 风险 3：不要一开始就引入过重 AST

当前阶段先做“协议 + 元数据 + 归一化”最合适。
直接上消息块 AST 虽然更强，但改动面更大，不适合作为第一步。

---

## 13. 本文结论

结论明确如下：

1. 当前渲染问题不能继续只靠前端判断。
2. 后端必须承担“回答协议化”和“结果归一化”的职责。
3. 前端后续应主要按协议渲染，而不是做大规模语义猜测。
4. 第一版最值钱的工作是：
   1. 扩展响应协议
   2. 引入回答归一化器
   3. 前端优先使用后端渲染元数据

这套方案能把消息渲染从“前端猜测驱动”，迁移到“后端协议驱动”，也更适合后续继续往生产级系统演进。

## 14. Phase 5 落地记录（2026-04-25）

本阶段解决的是前一阶段遗留的“artifact 只能展示，不能真正操作”的问题。

### 14.1 新增 workspace 文件预览接口

后端新增平台层只读接口：

1. `GET /api/sessions/{session_id}/workspace-files/preview`
2. 参数：
   1. `path`
   2. `max_chars`
3. 约束：
   1. 只允许读取当前会话 `workspace` 内文件
   2. 禁止绝对路径
   3. 禁止路径穿越
4. 返回：
   1. `content`
   2. `size_bytes`
   3. `total_chars`
   4. `truncated`
   5. `answer_format`
   6. `render_hint`

设计原因：

1. workspace 文件预览属于平台能力，不应该走 tool loop。
2. 用户点击 artifact 看文件内容，不应该再触发一次 agent 推理。
3. 预览接口复用 `AnswerNormalizer`，这样前端拿到的仍然是统一渲染协议。

### 14.2 artifact 卡片支持直接查看 workspace 文件内容

Flutter 前端调整为：

1. `file_id:` 类型 artifact：
   1. 继续执行“激活文件”
   2. 激活成功后同步进入当前上下文
2. workspace 路径类型 artifact：
   1. 主按钮改为“查看内容”
   2. 次按钮保留“复制路径”
   3. 点击后弹出预览面板

### 14.3 激活文件后增加输入区高亮反馈

仅仅把文件加入 `active_file_ids` 不够，用户看不到结果。

本阶段补了一个轻量状态：

1. `recentActivatedFileId`
2. 激活成功后保留约 3 秒
3. 输入框上方的“已激活上下文”文件 chip 会用强调样式高亮

这样 artifact 点击后的反馈闭环就完整了：

1. 点击来源文件卡片
2. 文件进入 active context
3. 输入区文件 chip 高亮提示

### 14.4 当前边界

本阶段仍然有意保持边界清晰，没有做以下内容：

1. workspace 文件双击直接编辑
2. artifact 与调试面板双向跳转
3. 上传文件 artifact 的正文预览

这些属于下一阶段扩展，不应把当前这一步做成过重的文件工作台。

## 15. Phase 6 落地记录（2026-04-25）

本阶段处理的是“普通回答内容虽然能显示，但排版体验差、重点信息不够突出”的问题。

### 15.1 问题定义

前一阶段主要解决的是：

1. 消息渲染协议
2. Markdown / 源码 / 文件内容区分
3. artifact 可操作化

但普通文本回答仍有两个明显问题：

1. 模型容易把一段自然说明拆成多行，前端显示会显得碎。
2. 关键结论、风险词、命令、文件名没有轻量强调时，阅读层级不够清楚。

### 15.2 第一版策略

本阶段刻意不引入新的协议字段，先做低风险版本：

1. 提示词层
   1. 普通回答使用自然段落
   2. 不要每句单独换行
   3. 可克制使用 `**重点词**`
   4. 可克制使用 `*次级术语*`
   5. 命令、文件名、变量名使用 `` `code` ``
2. 后端层
   1. 对 `plain_text` 回答做软换行合并
   2. 保留真正的段落边界
   3. 保留列表型内容的换行
3. 前端层
   1. 普通回答不再直接走单一 `SelectableText`
   2. 改为按段落渲染
   3. 支持轻量行内样式：
      1. `**bold**`
      2. `*italic*`
      3. `` `code` ``

### 15.3 为什么这一版不做下划线

下划线不属于 Markdown 的稳定基础子集，模型输出也容易不一致。

因此第一版刻意只支持：

1. 加粗
2. 斜体
3. 行内代码

下划线如果后面要做，应单独定义受控语法，而不是让模型自由发挥。

### 15.4 当前收益

这一步完成后：

1. 普通回答的段落会更自然。
2. 关键短语可以被模型用轻量强调语法标出来。
3. 前端不需要把普通回答也全部当完整 Markdown 处理。
4. 普通回答的阅读体验会明显优于“原样逐行吐字符串”。

## 16. Phase 7 落地记录（2026-04-25）

本阶段把普通回答进一步协议化，不再只靠前端局部样式优化。

### 16.1 新增 `layout_hint`

后端为回答增加 `layout_hint`，当前支持：

1. `brief`
2. `paragraph`
3. `bullets`
4. `steps`

传播链路已打通到：

1. `/api/chat`
2. SSE `answer_meta`
3. SSE `done`
4. 历史消息接口
5. workspace 文件预览接口

### 16.2 分类策略

当前策略刻意保持保守：

1. 短且不分段的普通回答 -> `brief`
2. 多段普通解释 -> `paragraph`
3. 以无序要点为主 -> `bullets`
4. 以编号步骤为主 -> `steps`
5. 真正的 Markdown 文档 / 源码 / code block 仍然走原有 `render_hint`

关键调整点：

1. 轻量强调（`**bold**` / `*italic*` / `` `code` ``）不再单独把回答升级成完整 Markdown 文档。
2. 普通步骤和普通要点列表，也不再一律升格为 Markdown 文档。

这样普通回答可以保留“普通回答协议”，同时继续拥有轻量格式能力。

### 16.3 Flutter 渲染策略

Flutter 侧对 plain renderer 做了正式分流：

1. `brief`
   1. 更紧凑的短答样式
   2. 轻度强调
2. `paragraph`
   1. 按段落渲染
   2. 保留轻量行内样式
3. `bullets`
   1. 真正的要点列表样式
4. `steps`
   1. 真正的步骤列表样式

### 16.4 当前意义

这一阶段完成后，普通回答不再只是：

1. 原样字符串
2. 或者被迫升级成完整 Markdown

而是形成了独立的“普通回答布局协议”，这对后续商业级对话体验是必要的基础层。

## 17. Phase 8 落地记录（2026-04-25）

本阶段继续解决“普通回答虽然能读，但还是像一大块文本”的问题，重点不是继续堆 Markdown 能力，而是把结构化 plain answer 做成更明确的阅读块。

### 17.1 正式消息区移除工具芯片

正式 assistant 消息区不再直接展示 `memory_search`、`workspace_write_file` 这类工具芯片。

原因：

1. 这类信息属于执行过程或调试信息，不属于最终面向用户的正文内容。
2. 保留在正式消息区会打断阅读节奏，破坏页面层次。

当前策略：

1. 工具调用信息继续保留在执行过程 / 调试路径。
2. 正式回答只展示正文和与正文相关的 artifact。

### 17.2 结构化普通回答自动分组

Flutter 侧为 plain answer 增加了结构化分组解析器，当前重点覆盖这类场景：

1. 多方案推荐
2. 多路线对比
3. 多选项建议
4. 统一的通用建议
5. 结尾的追问 / CTA

解析后会自动拆成：

1. 引导段落
2. 多个 section card
3. 字段行（例如 `亮点 / 行程建议 / 贴士`）
4. 通用建议列表
5. 结尾 CTA 卡片

### 17.3 当前收益

这一阶段完成后：

1. 多方案型普通回答不再只是一个长文本块。
2. 方案标题、字段标签、结尾追问都会有更明确的视觉层级。
3. 用户扫读信息的成本会明显下降。
