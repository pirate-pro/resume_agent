# 单 Agent Runtime 最小实现文档

## 1. 文档目标

本文档用于指导 AI 直接生成一个 **可运行的单 Agent 骨架项目**。

项目目标：

* 使用 **FastAPI** 提供 HTTP API
* 使用 **uv** 管理环境与依赖
* 不使用数据库
* 使用 **JSONL + Markdown + 本地文件目录** 作为最小存储方案
* 实现一个最小可运行的单 Agent Runtime
* Runtime 需具备：

  * session / runtime
  * context assembling
  * memory
  * skill
  * tool
* 强调 **模块化**、**职责清晰**、**最小可运行**
* 为后续扩展 DeerFlow 风格 harness / OpenClaw 风格 subAgent/session orchestration 预留空间

该项目当前只追求：

> **先跑通一个可用的骨架，不追求复杂优化。**

---

## 2. 非目标

当前版本不实现以下能力：

* 多 Agent 编排
* lead agent / subAgent
* 向量数据库
* embedding 检索
* 上下文压缩与长上下文总结
* 后台任务队列
* WebSocket / SSE 流式输出
* 鉴权系统
* 权限系统
* 数据库持久化
* 分布式锁
* 多 worker 并发写文件安全

---

## 3. 技术约束

### 3.1 技术栈

* Python 3.12+
* FastAPI
* Pydantic v2
* httpx
* uvicorn
* uv
* pytest
* mypy

### 3.2 编码风格约束

生成代码时必须遵守以下规则：

1. 内部模型优先使用 `@dataclass(slots=True)`
2. 外部 API 请求响应模型使用 `Pydantic BaseModel`
3. 抽象接口使用 `Protocol`，不要使用 ABC
4. 除入口文件外，业务逻辑全部封装为 class
5. 所有属性默认私有，使用 `_xxx` 命名
6. 对外通过方法暴露能力，不直接暴露内部属性
7. 每个函数入口必须做参数合法性检查（不仅仅依赖类型）
8. 类名使用 CamelCase，变量与方法使用 snake_case
9. 每个模块必须写对应测试
10. 所有模块必须显式声明 `__all__`
11. 代码尽量最小化，不做无必要抽象
12. 单进程、单 worker 运行

### 3.3 运行约束

服务启动方式默认：

```bash
uv run uvicorn app.main:app --reload --workers 1
```

必须明确：

* 当前版本仅支持单进程
* 当前版本通过内存锁管理 session 内串行执行
* 不保证多进程写文件安全

## 4. 项目目标产物

生成的项目必须最终满足：

1. 可以启动 FastAPI 服务
2. 提供 `POST /api/chat` 接口
3. 支持新建 session 或使用已有 session
4. 能将用户消息写入 session 事件日志
5. 能加载 skills Markdown
6. 能从 memory 中检索相关内容
7. 能注册与执行工具
8. 能完成最小工具调用循环
9. 能将 assistant 回复写入日志
10. 能通过接口查看 session 事件
11. 能通过接口查看 memory
12. 测试可运行

---

## 5. 总体架构

采用四层结构：

### 5.1 API 层

职责：

* 接收 HTTP 请求
* 解析请求参数
* 调用 service 层
* 返回 HTTP 响应

禁止：

* 直接访问文件存储
* 直接调用模型
* 编排 runtime 细节

### 5.2 Service 层

职责：

* 用例编排
* 协调 session 加载/创建
* 调用 runtime
* 将 runtime 输出转换为响应 DTO

### 5.3 Runtime 层

职责：

* 组装上下文
* 调用模型
* 处理工具调用
* 记录事件
* 读写 memory
* 产出最终回复

### 5.4 Infra 层

职责：

* JSONL 文件存储
* Markdown skill 加载
* 模型客户端实现
* session 锁
* 工具执行底层实现

## 7. 数据存储结构

项目根目录下使用：

```text
data/
├── sessions/
│   └── {session_id}/
│       ├── metadata.json
│       ├── events.jsonl
│       └── workspace/
└── memory/
    └── memories.jsonl
```

### 7.1 `sessions/{session_id}/metadata.json`

示例：

```json
{
  "session_id": "sess_xxx",
  "title": "New Session",
  "created_at": "2026-04-20T12:00:00Z",
  "updated_at": "2026-04-20T12:00:00Z"
}
```

### 7.2 `sessions/{session_id}/events.jsonl`

每行一个事件。

事件类型允许：

* `run_started`
* `user_message`
* `tool_call`
* `tool_result`
* `assistant_message`
* `memory_write`
* `run_finished`

示例：

```json
{"event_id":"evt_001","session_id":"sess_1","type":"user_message","payload":{"content":"帮我记住暂时不用数据库"},"created_at":"2026-04-20T12:01:00Z"}
{"event_id":"evt_002","session_id":"sess_1","type":"assistant_message","payload":{"content":"好的，我会记住你当前先不使用数据库。"},"created_at":"2026-04-20T12:01:05Z"}
```

### 7.3 `memory/memories.jsonl`

示例：

```json
{"memory_id":"mem_001","session_id":"sess_1","content":"用户当前先不使用数据库，使用 JSONL + Markdown 文件存储。","tags":["preference","storage"],"created_at":"2026-04-20T12:01:03Z","source_event_id":"evt_001"}
```

## 8. 核心接口设计原则

### 8.1 内外分离

* 对外 HTTP 层：Pydantic
* 对内运行对象：dataclass
* 依赖抽象：Protocol

### 8.2 最小抽象原则

必须有抽象的地方：

* 模型客户端
* session 存储
* memory 存储
* skill 存储
* 工具执行

不需要额外抽象的地方：

* 业务 service 可直接依赖 protocol
* 运行时不再套多余 facade

---

## 9. Pydantic 接口模型

文件：`app/schemas/chat.py`

需要定义以下模型：

### 9.1 `ChatRequest`

字段：

* `session_id: str | None = None`
* `message: str`
* `skill_names: list[str] = []`
* `max_tool_rounds: int = 3`

校验要求：

### 9.2 `ToolCallView`

* `arguments: dict[str, Any]`

### 9.3 `MemoryView`

字段：

* `memory_id: str`
* `content: str`
* `tags: list[str]`

### 9.4 `ChatResponse`

字段：

* `session_id: str`
* `answer: str`
* `tool_calls: list[ToolCallView]`
* `memory_hits: list[MemoryView]`

## 12. Runtime 核心职责

### 12.1 `SessionManager`

文件：`app/runtime/session_manager.py`

职责：

* 获取或创建 session
* 获取最近事件
* 获取 workspace 路径

要求：

* 对不存在 session 自动创建
* 负责基本参数校验

### 12.2 `EventRecorder`

文件：`app/runtime/event_recorder.py`

职责：

* 统一写入事件日志

要求：

* 生成 event_id
* 生成标准事件对象
* 调用 `SessionRepository.append_event`

### 12.3 `MemoryManager`

文件：`app/runtime/memory_manager.py`

职责：

* 写 memory
* 搜 memory
* 提供简单记忆策略

第一版策略：

* 支持显式工具写入 `memory_write`
* 支持基于 query 的简单全文检索
* 不做 embedding
* 不做自动总结压缩

### 12.4 `ContextAssembler`

文件：`app/runtime/context_assembler.py`

职责：

* 加载 skills
* 检索 memory
* 读取最近事件
* 组装 system prompt
* 生成 messages
* 输出 `ContextBundle`

上下文规则：

1. 固定基础 system prompt
2. 拼接 skills 文本
3. 拼接检索到的 memory 摘要
4. 读取最近 12 条用户/助手消息相关事件
5. 生成最终消息数组

第一版不做摘要压缩，仅做窗口截断。

### 12.5 `AgentRuntime`

文件：`app/runtime/agent_runtime.py`

职责：

执行一次完整 Agent Run。

执行流程：

1. 记录 `run_started`
2. 记录 `user_message`
3. 由 `ContextAssembler` 组装上下文
4. 调用模型
5. 如果模型返回 tool_calls，则执行工具
6. 记录 `tool_call` 和 `tool_result`
7. 将工具结果加入消息上下文，再次调用模型
8. 最终得到回复内容
9. 记录 `assistant_message`
10. 记录 `run_finished`
11. 返回 `AgentRunOutput`

约束：

* 最多允许循环 `max_tool_rounds`
* 若超过轮数仍持续请求工具，则终止并返回错误说明文本
* 工具失败时，不得伪造成功结果

## 13. Tool 设计

### 13.1 Tool 基础要求

文件：`app/tools/base.py`

定义最小工具接口。

每个工具必须是 class，并至少具备：

* `definition()`：返回 `ToolDefinition`
* `execute(arguments: dict[str, Any], session_id: str) -> ToolExecutionResult`

### 13.2 `ToolRegistry`

文件：`app/tools/registry.py`

职责：

* 注册工具
* 输出工具 schema
* 按名称执行工具

要求：

* 名称重复时报错
* 执行不存在工具时报错
* 工具参数必须做基本合法性校验

### 13.3 第一版内置工具

文件：`app/tools/builtins.py`

必须实现以下 4 个工具：

#### 1）`memory_write`

参数：

* `content: str`
* `tags: list[str] = []`

行为：

* 写入全局 memory

#### 2）`memory_search`

参数：

* `query: str`
* `limit: int = 5`

行为：

* 在 memory 中简单检索

#### 3）`workspace_write_file`

参数：

* `path: str`
* `content: str`

行为：

* 写文件到当前 session workspace
* 禁止路径逃逸

#### 4）`workspace_read_file`

参数：

* `path: str`

行为：

* 从当前 session workspace 读取文件
* 禁止路径逃逸

## 14. Skill 设计

技能文件位于：`app/skills/`

第一版必须提供以下三个 skill 文件：

### 14.1 `base.md`

内容要求：

* 说明 Agent 的基础行为
* 不得伪造工具结果
* 优先依赖工具而不是编造外部状态
* 对未知信息保持诚实

### 14.2 `memory.md`

内容要求：

* 何时写入 memory
* 何时检索 memory
* 不要把短期上下文误当长期偏好

### 14.3 `tools.md`

内容要求：

* 四个工具的作用
* 参数使用要求
* 工具失败时的处理规则

`SkillRepository` 只需要从 Markdown 文件读取文本即可，不需要复杂索引。
